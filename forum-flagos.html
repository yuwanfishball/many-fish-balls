▶ 05:00 – 26:20 Triton-TLE：不烧脑，也能写出高性能 SparseMLA —

本环节系统演示了 Triton-TLE（Triton Language Extension）如何显著降低高性能 SparseMLA（稀疏多头潜在注意力）算子的开发门槛：

  • SparseMLA 的计算特征：SparseMLA 是长上下文推理的核心算子，通过稀疏化注意力矩阵来减少计算量。其稀疏模式因模型结构和输入不同而变化，传统手工编写高性能 Kernel 需针对每种稀疏模式逐一调优，工程成本极高
  • Triton-TLE 的简化路径：Triton-TLE 提供了声明式稀疏描述原语，开发者只需描述稀疏模式的结构特征（如分块大小、稀疏比例），框架即可自动生成优化的分块策略与内存访问模式
  • 性能数据：使用 Triton-TLE 编写的 SparseMLA Kernel，在典型稀疏度（50%–90%）下达到手工优化版本 95% 以上的吞吐，开发时间从数天缩短至数小时
  • 适用边界：该方案不仅适用于 SparseMLA，还可推广至稀疏注意力、稀疏 MoE 等同类场景

核心要点

Triton-TLE 通过声明式稀疏编程原语，将高性能 SparseMLA 的开发从专家级手工优化转变为框架自动化流程，在保持近手工级性能的同时大幅降低工程复杂度。
▶ 26:20 – 53:49 FlagOS 在 Arm64 CPU 上的使能与性能优化：基于 Triton/TLE 的端侧 AI 实践 —

本环节展示了 FlagOS 在 Arm64 架构上的完整使能方案，将端侧 AI 推理从依赖 GPU 的单一范式拓展至 Arm64 CPU 平台：

  • Arm64 使能策略：FlagOS 利用 Triton/TLE 的跨架构编译能力，将同一套算子描述编译为 Arm64 NEON/SVE 向量指令，实现与 CUDA 后端共享上层代码、底层自动适配
  • 端侧性能优化手段：通过内存布局重排、算子融合和量化感知编译三项技术组合，在苹果 M 系列和高通骁龙 X Elite 等 Arm64 平台上实现大模型推理延迟降低 40%–60%
  • Triton Arm64 后端扩展：FlagOS 扩展了 Triton 编译器的 Arm64 代码生成路径，补充了 SVE 向量化策略与 Neon 指令映射表，使 Triton Kernel 可直接在 Arm64 平台上编译运行
  • 实测效果：在 M3 Max MacBook 上运行 7B 模型的推理吞吐达到 28 tokens/s，满足实时交互需求

核心要点

FlagOS 通过 Triton/TLE 跨架构编译能力，使同一套算子代码可同时编译为 CUDA 和 Arm64 后端，让端侧大模型推理摆脱对 GPU 的单一依赖，在 Arm64 平台上实现可量产部署的推理性能。
▶ 53:49 – 1:26:41 从算子全覆盖到性能突破：FlagOS 如何打通大模型推理优化最后一公里 —

本环节聚焦大模型推理优化中算子覆盖率与极致性能的平衡问题，展示了 FlagOS 的全算子覆盖方案：

  • 问题定位：主流推理框架在常见算子上性能优异，但在长尾算子（如非标准 Attention 变体、动态形状 Reshape、稀疏 MoE 路由等）上往往回退到性能低下的通用实现。这些长尾算子虽在总计算量中占比不大，却常成为端到端延迟的瓶颈
  • FlagOS 全覆盖策略：通过自动算子搜索（Auto-Tuning）结合模板匹配技术，对模型计算图中每一个算子——无论主算子和长尾算子——都生成芯片特定的优化实现，消除任何性能回退路径
  • 实测数据：在 LLaMA-3-70B 推理场景中，全算子优化相比仅优化主算子的方案，端到端延迟降低 18%，首 Token 生成时间缩短 25%
  • 使用方式：整个过程从模型导入到优化算子生成，仅需一条命令触发，无需人工介入

核心要点

大模型推理的性能瓶颈往往不在主算子而在长尾算子。FlagOS 的自动算子搜索与全覆盖策略，确保计算图中每一个算子都获得芯片级优化实现，真正打通了从模型到部署的"最后一公里"。
▶ 1:26:41 – 1:57:45 打破 AI 芯片孤岛：统一编译器与公共中间表示层的构建之路 —

本环节深入阐述了 FlagOS 统一编译器架构的技术路线，旨在解决 AI 芯片生态碎片化的核心痛点：

  • 芯片碎片化现状：当前市面上有超过 30 种 AI 加速芯片（NVIDIA GPU、AMD GPU、华为昇腾、寒武纪、Graphcore 等），每种芯片都有独立的编译工具链和算子库，开发者需为每种芯片单独编写和优化代码，迁移成本极高
  • 统一中间表示层（UIR）设计：FlagOS 采用分层中间表示架构——上层为芯片无关的计算图 IR，中层为可配置的张量运算原语 IR，底层为芯片特定的指令映射层。开发者只需针对上层 IR 编写一次代码，编译器的后端插件体系通过规则引擎和性能模型自动将统一 IR 映射为各芯片的最优指令序列
  • 当前适配进展：已适配 NVIDIA、AMD、华为昇腾、寒武纪四家主流芯片，社区贡献者正在扩展至更多国产芯片平台

核心要点

FlagOS 的统一编译器架构通过分层中间表示和插件式后端映射，将"写一次代码、多芯片运行"从愿景变为工程现实，是打破 AI 芯片生态碎片化的关键基础设施。
▶ 1:57:45 – 2:16:02 量智融合：FlagQuantum 如何用量子计算加速 AI 训练与推理 —

本环节展示了 FlagQuantum 子项目在量子-经典计算融合方向的最新进展，阐述量子计算为 AI 训练与推理提供加速的可行路径：

  • 量子加速 AI 的三个可行方向：当前量子计算在组合优化（如 MoE 专家路由选择）、线性代数加速（如量子态内积计算）和高维概率采样三个方向上已展现出超越经典计算的潜力，这些方向与大模型的计算需求高度契合
  • FlagQuantum 混合架构：采用量子-经典混合计算范式，经典 GPU/CPU 负责主体计算，量子处理单元（QPU）负责特定子任务加速，两者通过 FlagOS 统一调度层无缝协同
  • 验证结果：在 MoE 模型的专家路由优化中，量子退火算法相比经典贪心算法在专家利用率上提升 12%，在 16 专家 × 4 选择的配置下路由延迟降低 30%
  • 现实约束：量子硬件仍处于中等规模含噪声（NISQ）阶段，实际收益受限于量子比特数量和门保真度，在可预见的近期以经典计算为主、量子计算为辅的混合模式运行

核心要点

FlagQuantum 通过量子-经典混合计算架构，在 MoE 路由优化等特定 AI 子任务上验证了量子加速的可行性。提前布局量子-经典融合计算框架，为 AI 算力开辟了超越经典摩尔定律的新增长曲线。
▶ 2:16:02 – 2:37:08 Agent4Agent：多智能体互操作技术框架 —

本环节提出了 Agent4Agent（A4A）多智能体互操作技术框架，解决 AI Agent 之间的协作与通信问题：

  • 多智能体协作的挑战：当多个 AI Agent（如代码生成 Agent、推理优化 Agent、部署 Agent）需要协同完成一个端到端任务时，面临接口不统一、上下文传递断裂、任务依赖关系管理复杂三个核心问题
  • A4A 框架设计：定义了标准化的 Agent 间通信协议（Agent Communication Protocol），包括任务描述格式、中间结果交换规范和状态同步机制。每个 Agent 只需实现 A4A 标准接口即可接入多智能体协作网络
  • 任务编排引擎：框架内置了基于有向无环图（DAG）的任务编排器，自动解析 Agent 间的依赖关系、调度并行执行路径并处理异常回退，无需开发者手动编写编排逻辑
  • 与 FlagOS 的集成：A4A 框架已集成至 FlagOS 插件体系，KernelGen、FlagScale Agent、vLLM-Plugin-FL 等工具可通过 A4A 协议互操作，形成从算子生成到模型部署的完整 Agent 协作链路

核心要点

Agent4Agent 框架通过标准化 Agent 间通信协议和 DAG 任务编排引擎，解决了多智能体协作中的接口统一和依赖管理问题。与 FlagOS 插件体系集成后，实现了从算子生成到模型部署的多 Agent 自动化协作链路。
▶ 2:37:08 – 3:11:19 从"像素级调优"到"一句话生成"：KernelGen 如何重塑 AI 算子开发 —

本环节演示了 KernelGen——一个基于大语言模型的 AI 算子自动生成工具，呈现从传统手工调优到自然语言驱动生成的范式转变：

  • 传统算子开发的瓶颈：编写一个高性能 GPU Kernel 需要开发者同时掌握 CUDA/Triton 编程、GPU 微架构特性（共享内存、寄存器分配、Warp 调度）和目标芯片的优化技巧，一个复杂算子的开发周期通常以周为单位
  • KernelGen 工作流程：开发者用自然语言描述算子的计算逻辑（如"实现一个支持 GQA 的 Flash Attention 前向 Kernel，head_dim=128，使用 FP16"），KernelGen 自动生成优化的 Triton Kernel 代码，并附带性能基准测试和正确性验证结果
  • 三层验证流水线：生成代码依次通过形式化正确性检查、小规模张量数值对比验证和目标芯片完整性能基准测试，三项全部通过后输出最终代码
  • 实测效果：对 50 个常见算子的测试显示，KernelGen 生成的代码在 85% 的案例中达到或超过手工优化版本的性能，剩余 15% 经人工微调后也可达标。每次生成结果记录至结构化知识库，使生成质量随使用次数持续提升

核心要点

KernelGen 将 AI 算子开发从"像素级手工调优"转变为"自然语言描述 + 自动验证"的高效流程，在 85% 的常见算子上达到手工级性能。这一工具显著降低了算子开发的门槛和周期。
▶ 3:11:19 – 3:43:06 大模型训推迁移的"无人驾驶"时代：FlagScale Agent 系统级创新与实践 —

本环节展示了 FlagScale Agent——一个面向大模型训练与推理迁移的系统级自动化工具，标志着训推迁移从手动工程进入"无人驾驶"时代：

  • 训推迁移的核心难点：大模型从训练集群迁移到推理集群（或跨芯片迁移）时，需处理分布式策略重配置、模型切分方式重构、精度对齐、通信拓扑适配等一系列复杂问题，传统方式依赖资深工程师逐一排查和调优
  • "无人驾驶"架构：借鉴自动驾驶分级概念（L1-L5），FlagScale Agent 提供从辅助建议（L2）到全自动执行（L4）的多级自动化模式。核心由四个模块构成——感知模块扫描源平台配置、规划模块生成迁移策略、执行模块按步骤实施迁移、验证模块自动运行正确性和性能测试
  • 自动故障恢复：当迁移过程中出现精度不匹配或性能回退时，Agent 自动回溯至上一个正确检查点，调整策略后重新执行，整个过程无需人工介入
  • 规模化验证：已在超过 100 个模型（涵盖 LLaMA、Qwen、DeepSeek 等主流架构）和 5 种芯片平台的交叉组合中完成验证，迁移成功率达到 96%

核心要点

FlagScale Agent 实现了大模型训推迁移的全流程自动化闭环——从源平台感知、策略规划、执行到正确性验证。在 100+ 模型 × 5 芯片平台的规模验证中达到 96% 成功率，标志着跨平台迁移从专家手工工程走向工业化自动流程。
▶ 3:43:06 – 4:13:06 以 Skill 封装工程经验：让 AI Agent 接管 vLLM-plugin-f 自动化部署 —

本环节展示了基于 Skill 封装理念的 vLLM-Plugin-FL 自动化部署方案，将工程经验固化为可被 AI Agent 调用的标准化技能模块：

  • Skill 封装理念：将模型部署中的常见工程经验——包括芯片适配逻辑、算子替换策略、性能配置模板——封装为独立的"Skill"模块。每个 Skill 是一个自包含的、可被 Agent 发现和调用的能力单元，内部封装了特定领域的工程知识
  • Agent 接管流程：AI Agent 接收用户的高层意图（如"将 Qwen2-72B 部署至华为昇腾 910B，4 卡 TP"），自动匹配所需的 Skill 组合（芯片适配 Skill + TP 切分 Skill + 性能调优 Skill），按依赖顺序编排执行，并在每个环节完成后进行验证
  • 降低门槛的效果：在演示中，Agent 在 3 分钟内完成了模型格式转换、算子适配、分布式策略配置和推理服务启动的全流程，将传统需要数小时专家手工操作的任务简化为一次自然语言指令
  • Skill 生态扩展性：社区开发者可贡献新的 Skill 模块至 Skill 市场，Agent 自动发现并学习使用新 Skill，使部署能力随社区贡献持续扩展

核心要点

Skill 封装机制将分散的工程经验转化为标准化的、可被 Agent 调用的能力模块。AI Agent 通过自动匹配和编排 Skill 组合，将跨芯片模型部署从数小时的手工操作简化为一次自然语言指令，这是 FlagOS 开放计算生态走向大规模普及的关键一步。
